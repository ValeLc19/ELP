ELP · Events pipeline

How the app refills its own events every day

Nobody types in El Paso's events by hand. Once a day, a scheduled job reads visitelpaso.com, keeps the upcoming in-person ones, and refreshes the catalog the app shows — cleaning up after itself so the list never goes stale or piles up. Here's the whole run, start to finish.

02

The daily run

scripts/sync_visitelpaso.py
1
A timer kicks it off.github/workflows/sync-events.yml

A GitHub Actions cron fires once a day (and on a manual click). It runs on GitHub's own machines — not the app's server — so a slow, multi-page crawl never risks a request timeout.

cron: dailyworkflow_dispatch
2
Find every event URL_fetch(SITEMAP)

It reads the sitemap and pulls out every /events/<slug> link. Slugs that already carry a date outside the next 90 days get dropped right here — no page fetch wasted on them.

sitemap.xmlwindow: today → +90d
3
Read each event page_event_node(html)

For each surviving URL it fetches the page — throttled to ~0.35s apart to stay polite, capped at 250 per run — and lifts the JSON-LD Event block: title, date, address, price, description.

schema.org/Eventthrottle 0.35scap 250
4
Keep only what belongsthe filter gate →

Each event has to clear the gate below: upcoming, in-person, and in El Paso. Online talks, conferences, and anything out of window are dropped. Recurring series are capped at 3 instances so a daily museum exhibit can't flood the list.

upcoming + localonline / conference≤ 3 per series
5
Dress it upguess_category · pick_theme

Keyword rules sort each event into one of the app's six categories and pick a matching Unsplash stock image (the app's own art, never visitelpaso's copyrighted photos). Price becomes Free or See details; family-friendly events get flagged.

Food · Arts · MusicSports · Outdoors · Marketsthemed image
6
Put it on the map_geocode_addr(addr)

The address is turned into map coordinates via Nominatim (1 req/sec). It tries the street first, then the venue name, then the whole string — so a venue-only address still lands right instead of dropping to the city center. Only new or mislocated rows are geocoded; settled ones are left alone.

OpenStreetMapstreet → venue → citynew rows only
7
Save & sweepdb.commit()

Each event is upserted by slug — new ones inserted, existing ones refreshed in place. Then the job prunes its own rows that have passed or vanished from the sitemap. Curated seed events are never touched.

upsert visitelpaso rowsprune past / removedseed untouched
03

What clears the gate

stage 4, up close

Kept

  • ✓Starts today through +90 days
  • ✓Has a physical El Paso address (matches “el paso”, “, TX”, or “texas”)
  • ✓Carries real JSON-LD event data on the page
  • ✓Is one of the first 3 instances of its series

Dropped

  • ×Online / virtual — webinar, livestream, zoom, webcast
  • ×Conference-type — summit, congress, expo, convention
  • ×Past, or beyond the 90-day window
  • ×No address, not local, or the 4th+ repeat of a series
04

Why it never makes a mess

Idempotent

Run it once or ten times — same result. Events key on their slug, so a re-run refreshes rows instead of duplicating them.

Self-pruning

It only ever deletes rows it owns (source='visitelpaso') once they're past or gone upstream. Hand-curated seed events stay put.

Fail-safe

If the API is unreachable, the app quietly serves its bundled event snapshot — so the page always loads, sync or no sync.
